# Bug 012: Test 10.3 — `SEE /` output format does not match test expectation

## Test

**10.3** — `SEE /`
Got (contains check fails): `0x20000dd0: <forth>\n0x20000dd0: /MOD\n0x20000dd4: SWAP\n0x20000dd8: DROP`
Expected substring: `/MOD SWAP DROP`

## Summary

The test uses `"match": "contains"` and checks for the substring `/MOD SWAP DROP`.
`SEE` outputs each word on a **separate line** prefixed with its address:

```
0x20000dd0: <forth>
0x20000dd0: /MOD
0x20000dd4: SWAP
0x20000dd8: DROP
```

The words `/MOD`, `SWAP`, and `DROP` are all present but separated by newlines
and addresses, so the substring `/MOD SWAP DROP` never appears.

## Options

**Option A — fix the test** (test suite change):
Use a regex match that allows for the words to appear in any whitespace-separated format:

```python
"match": "regex",
"expect": r"/MOD.*SWAP.*DROP",
```

**Option B — fix `SEE`** (firmware change):
Emit Forth-defined word bodies on a single line without addresses:
```
/ : /MOD SWAP DROP
```

## Severity

Test suite / format mismatch only — `SEE` decompiles correctly; the content is
right but the format differs from the test's assumption.
